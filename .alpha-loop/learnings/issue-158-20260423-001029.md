## What Worked
- Schema-only scope was respected: types, parser, and defaults added without touching runtime, keeping the PR focused and backwards-compatible
- Parser validates endpoint references, stage names, endpoint types, and fallback modes with warn-and-skip semantics instead of throwing, matching the "no routing => no change" requirement
- Unit tests covered all acceptance criteria (default, profiles, invalid stage, unknown endpoint, malformed YAML); build and tests passed on first try

## What Failed
- Initial implementation included a dead `_issueId` parameter on `resolveRoutingStage` with a misleading JSDoc; caught and fixed in review
- One acceptance criterion ("`run --once` honors per-stage routing and logs which model ran") is not satisfied by this PR — correctly deferred to sibling sub-issues #159/#160, but the AC wording on #158 still implies runtime wiring

## Patterns
- Warn-and-skip parsing for optional nested config blocks: invalid sub-fields log a `[config]` warning and are dropped, rather than failing the whole load — preserves forward-compat with future schema additions
- Cross-reference validation inside the parser (stage.endpoint must exist in routing.endpoints) catches typos at config-load time instead of at stage-dispatch time
- `const VALID_X: readonly X[] = [...] as const` arrays paired with `.includes()` give runtime validation that stays in sync with the union type

## Anti-Patterns
- Carrying unused parameters forward "for future wiring" — adds a false API surface and misleads readers; add the parameter when the caller actually needs it
- JSDoc that describes intended behavior of a sibling function rather than the function it's attached to — drifts immediately and misleads reviewers
- Splitting an epic's acceptance criteria across sub-issues without rewording the child issue's own ACs to match its actual scope — creates the illusion of unmet requirements
