- Backend primitives (resolveRoutingStage, buildEndpointEnv, checkStagePrerequisites) were cleanly separated and fully unit-tested before wiring into callers
- Reviewer caught a critical doc/code mismatch where single-agent `lmstudio`/`ollama` mode dispatched to claude/codex CLI without injecting base URL env vars

## What Failed
- Initial implementation shipped docs claiming env vars were auto-set in single-agent local mode but the code did not do this — silent misrouting to cloud APIs
- Per-stage routing primitives are unwired: no caller in `pipeline.ts`, `run.ts`, or `review.ts` invokes them, so a `routing:` block is parsed but inert (acceptable per epic #165 breakdown — deferred to #160/#161)

## Patterns
- When adding agent types that wrap existing CLIs, auto-inject endpoint env vars at the `spawnAgent` boundary, not at config load — keeps user shell exports as the source of truth
- Restrict preflight HTTP probes to loopback addresses (`localhost`, `127.0.0.1`, `::1`, `*.local`) to avoid spurious network calls for remote endpoints
- Test env-var precedence explicitly: (1) auto-injection when unset, (2) user export wins, (3) `options.env` override wins

## Anti-Patterns
- Don't write docs that promise behavior the code doesn't implement — silent misrouting to a paid cloud API is a high-blast-radius bug
- Don't ship "primitive + caller" features in one issue without verifying the caller actually invokes the primitive — config that parses but does nothing is worse than a parse error

## Suggested Skill Updates
- Add to `aai-core:code-review` or reviewer agent prompt: "When adding a new agent/backend type, grep all spawn/dispatch sites to confirm the new type is actually handled, not just declared in config types"
- Add to `docs-sync` skill: "When docs claim env vars or behaviors are auto-applied, verify by tracing the code path from CLI entry to child-process spawn — don't trust the doc as evidence the wiring exists"
