## Patterns
- **Cost proxy for MVP**: Using duration×0.001 as a rough cost estimate when token counts aren't available yet is a reasonable bootstrapping approach — avoids blocking on full token tracking infrastructure
- **Pareto frontier as JSON**: Storing evaluated configs in a `pareto.json` with (score, cost, config_hash) tuples enables incremental search across sessions without re-running previous configs
- **Step-level eval cases as fixtures**: Structured `metadata.yaml` + `checks.yaml` + `input.md` pattern makes eval cases self-contained and easy to add

## Anti-Patterns
- **Duplicating config parsing logic**: `loadYamlOverrides` in eval.ts re-implements pipeline/pricing parsing already in config.ts — should share a single parser to avoid drift
- **Ignoring per-step overrides in eval runner**: The original `runStepEval` read `config.model` directly instead of `resolveStepConfig()`, silently defeating the entire purpose of per-step configuration search
