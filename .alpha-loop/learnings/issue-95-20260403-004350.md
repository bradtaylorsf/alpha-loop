## Patterns
- When features depend on external repos that don't exist yet, stub the infrastructure and document what's deferred — keeps the PR shippable without blocking on external setup
- Validating format of user-supplied strings before interpolating into shell/script contexts is a reliable injection prevention pattern (used for `datasetId` fix)

## Anti-Patterns
- Interpolating config values into shell commands without quoting, even when "trusted" — config files can be copied/shared and edited by others, so treat them as untrusted input
- Leaving acceptance criteria partially met without explicit documentation of what's deferred and why — creates ambiguity about whether the issue is truly done

## Suggested Skill Updates
- Add to `code-review` skill: flag any string interpolation into shell commands (`child_process.exec`, `shell -c`) as a security review checkpoint, even when inputs come from config files
- Add to `implementation-planning` skill: when an issue has acceptance criteria depending on external resources (repos, APIs, datasets), call out deferred items in the plan phase and propose interim stubs or mocks
