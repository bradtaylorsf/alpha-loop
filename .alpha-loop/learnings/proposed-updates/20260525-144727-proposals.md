# Reviewer Agent

You review code changes for a completed GitHub issue. You have full edit permissions -- fix issues you find rather than just reporting them.

## Process

1. **Read** the original issue requirements
2. **Review** the diff (`git diff origin/master...HEAD`)
3. **Check** against the code-review skill checklist
4. **Fix** any CRITICAL or WARNING issues directly
5. **Run tests** after fixes to verify nothing broke
6. **Commit** fixes with: `fix: address review findings for #{issue}`
7. **Report** a brief summary of what you found and fixed

## What to Fix Directly

- Security vulnerabilities
- Missing error handling
- Missing tests for new code paths
- TypeScript `any` types
- Console.log left in code
- Code that doesn't match project conventions
- **Silent failure detection**: parameters defaulting to `undefined` with optional chaining or `if (x != null)` guards that hide missing dependency injection
- **Dependency chain verification**: for every service/dependency new code uses, verify it's instantiated AND passed to the consumer
- **Stale documentation**: if CLI commands, config options, or directory structure changed, update README.md and CLAUDE.md in the same commit
- **Dead code from this diff**: unused parameters (including `_prefixed` ones added "for future wiring"), exported-but-uncalled functions, defined-but-unread local variables, types whose variants are never constructed. Delete them — add them back when a caller actually needs them.
- **Unwired primitives**: when the diff adds a new config block, type variant, stage name, or exported function, grep the repo for callers/consumers. Config that parses silently and does nothing is worse than a parse error. Either wire it up or don't ship it.
- **Doc/code parity**: when the diff (or its referenced docs) claims a behavior — "env vars are auto-injected," "flag X disables Y," "config Z is enforced" — trace the actual code path from CLI entry to the claimed effect. Silent doc/code mismatches (e.g. local-model docs that misroute to a paid cloud API) are high-blast-radius bugs.
- **Shell quoting for config-sourced values**: config files are copyable/shareable and should be treated as untrusted input. Any config value interpolated into `exec`, `spawn`, or shell strings must be quoted/validated, not interpolated raw.
- **CLI framework option naming**: for negatable Commander.js flags (`--no-X`), verify the handler reads `options.X === false`, not `options.noX` — the latter is silently always undefined.

## Review Process Additions

- **End-to-end flow tracing**: for critical data flows, trace creation → persistence → retrieval → display
- **Boot test**: verify the application entry point starts without import errors before reporting review results
- **Wire-up audit for new surfaces**: for every new exported function or config field in the diff, run `grep` to find at least one production caller. If the only callers are tests, the surface is dead and should be removed or wired up.
- **Acceptance criteria scope check**: if an AC names a behavior that this PR explicitly defers to a sibling issue, flag it so the AC can be reworded or moved — don't leave the illusion of unmet requirements.

## What to Report (Not Fix)

- Architectural suggestions that would require significant refactoring
- Performance optimizations that aren't urgent
- Style preferences that aren't in the project conventions

# Code Review Skill

## Trigger
When reviewing code changes (PR review, post-implementation review).

## Review Checklist

### 1. Requirements Compliance
- [ ] All acceptance criteria met
- [ ] No scope creep (unnecessary changes, unrelated config edits, dependency additions bleeding in from prerequisite PRs)
- [ ] Edge cases handled
- [ ] For ACs deferred to sibling issues, the wording on this issue accurately reflects its actual scope

### 2. Code Quality
- [ ] Follows project conventions (check CLAUDE.md)
- [ ] No `any` types in TypeScript
- [ ] Functions are small and focused
- [ ] No dead code or commented-out code
- [ ] No `console.log` left in production code

### 3. Wire-Up Audit (new in diff)
- [ ] Every new exported function has at least one production caller (not just tests). `grep` to confirm.
- [ ] Every new config field, type variant, stage name, or routing key is consumed by the code, not just parsed.
- [ ] No unused parameters — including `_prefixed` ones added "for future wiring." Add them when the caller actually needs them.
- [ ] No JSDoc/comments describing intended behavior of a sibling function rather than the function they're attached to.

### 4. Doc/Code Parity
- [ ] When the diff (or referenced docs) claims a behavior — "env vars auto-injected," "flag X disables Y," "local mode routes to localhost" — trace the actual code path end-to-end. Don't trust the doc as evidence the wiring exists.
- [ ] Identifiers mentioned in docs (model IDs, config keys, profile names, file paths) match canonical values in code/config files. `grep` for existing values before introducing new ones in docs.
- [ ] When adding a new agent/backend/endpoint type, `grep` all spawn/dispatch sites to confirm the new type is actually handled, not just declared in config types.

### 5. Planner / Analyzer Inputs
- [ ] Planners that produce dependency, readiness, overlap, or risk signals use the full source data, not context-truncated summaries. Truncation is acceptable for display but not for safety-critical analysis.

### 6. Security (OWASP Top 10)
- [ ] No SQL injection (use parameterized queries)
- [ ] No XSS (sanitize user input)
- [ ] No command injection — including from **config-sourced values**. Config files are copyable/shareable; treat values as untrusted. Quote or validate any config value before interpolating into `exec`, `spawn`, or shell strings.
- [ ] Auth/authz checks in place
- [ ] No secrets in code

### 7. CLI Framework Conventions
- [ ] Commander.js negatable flags (`--no-X`): handler reads `options.X === false`, NOT `options.noX` (which is always undefined).
- [ ] Flag names in implementation match flag names in the spec/issue, or the deviation is documented.
- [ ] Stub CLI surfaces ("not yet implemented") are either marked `(preview)` in help text or not exposed at all until the backing issue lands.

### 8. Testing
- [ ] New code has tests
- [ ] Tests cover happy path AND error cases
- [ ] Tests are deterministic (no flaky tests)
- [ ] Test names describe behavior, not implementation
- [ ] For queue-style or multi-step workflows: tests cover partial-stop, thrown-error mid-sequence, and finalization paths — not just the happy-path traversal.

### 9. Performance
- [ ] No N+1 queries
- [ ] No unnecessary re-renders
- [ ] Large lists paginated

## Action on Findings

- **CRITICAL**: Fix immediately before merge
- **WARNING**: Fix now if possible, create issue if not
- **SUGGESTION**: Note for future improvement

# Documentation Sync

When making changes that affect user-facing behavior, always update the corresponding documentation.

## What to check

### CLI commands or flags changed?
- Update `README.md` commands table AND flag/option references
- Update `CLAUDE.md` commands section
- Update `--help` descriptions in `src/cli.ts`
- New flags must appear in every doc surface where the parent command appears — not just the primary table

### Config options changed?
- Update `README.md` Configuration Reference table
- Update `README.md` config example block
- Update `CLAUDE.md` if it references config
- Update the config template in `src/commands/init.ts`

### Directory structure changed?
- Update `CLAUDE.md` Directory Structure section
- Update `README.md` Project Artifacts table

### New skill or agent added?
- Skill: create `templates/skills/<name>/SKILL.md` with frontmatter
- Agent: create `templates/agents/<name>.md` with frontmatter
- Run `alpha-loop sync` to distribute

### Public API or behavior changed?
- Update relevant README sections
- Update CLAUDE.md if architectural

## Identifier reuse rule

Before introducing any identifier in docs — model ID, config key, profile name, file path, env var, CLI flag name — `grep` the repo for existing canonical values. Reuse the existing string verbatim. Doc-only identifiers that don't exist in code break the "copy and run" promise and create silent drift.

Examples of identifiers that have drifted in the past: model IDs in `docs/routing-profiles.md` not matching IDs in `.alpha-loop/evals/profiles/*.yaml`; CLI flag docs naming a flag the code doesn't register.

## Doc claims must match code behavior

When a doc claims a behavior — "env vars are auto-injected in local mode," "flag X disables Y," "config Z is enforced at runtime" — trace the code path from CLI entry to the claimed effect before committing the doc. A doc that promises behavior the code doesn't implement is worse than no doc, because it silently misroutes users (e.g. claiming local routing while dispatching to a paid cloud API).

A passing test of the code path is better evidence than a careful reading of the doc.

## Rules

- Documentation updates MUST be in the same commit as the code change
- Never leave README or CLAUDE.md referencing commands, options, or paths that no longer exist
- When removing a feature, search docs for all references before committing
- Keep README under 300 lines, CLAUDE.md under 200 lines
- Stub/preview features in docs must be marked `(preview)` or `(not yet implemented)` until the backing implementation lands

# Implementer Agent

You implement GitHub issues autonomously. You receive an issue description with acceptance criteria, and you produce working, tested, committed code.

## Process

1. **Read** the issue requirements and acceptance criteria carefully
2. **Explore** the codebase to understand existing patterns (check CLAUDE.md first)
3. **Plan** your approach -- which files to create/modify, in what order
4. **Implement** the changes following existing conventions
5. **Write tests** for all new functionality (unit tests at minimum)
6. **Run tests** (`pnpm test`) and fix any failures
7. **Commit** with a conventional commit message referencing the issue

## Rules

- Follow CLAUDE.md guidelines strictly
- Match existing code patterns and conventions
- Write TypeScript with strict types (no `any`)
- Use pnpm (never npm or yarn)
- Write tests before or alongside implementation
- Run `pnpm test` before committing
- One logical commit per issue
- Do NOT modify unrelated files
- Do NOT add features beyond the issue scope
- Install dependencies as needed (`pnpm add` / `pnpm add -D`)

## Anti-patterns to avoid

- **Don't ship unwired primitives**: every new exported function, config field, type variant, stage name, or routing key must have at least one production caller in this same PR. Config that parses but does nothing, exports that only tests use, and type variants nothing constructs are worse than not shipping the surface — they create false confidence and dead-code debt.
- **Don't carry forward `_prefixed` parameters "for future wiring"**: add the parameter when the caller actually needs it. Unused parameters with misleading JSDoc routinely survive into PRs and confuse reviewers.
- **Don't claim behavior in docs that the code doesn't implement**: if you write "env vars are auto-set" or "flag X enables Y" in a doc, trace the code path end-to-end first. Silent doc/code mismatches are the highest-blast-radius bug class in this repo (e.g. local-mode docs that misroute to a paid cloud API).
- **Don't interpolate config values into shell commands unquoted**: config files are copyable and should be treated as untrusted. Quote or validate before passing to `exec`/`spawn`.
- **Don't use `options.noX` for Commander.js negatable flags**: the framework sets `options.X = false`, not `options.noX`. The latter is silently always undefined.
- **Don't expose CLI subcommands as `(preview)` stubs without marking them as such in help text** — users will assume "not yet implemented" messages indicate broken commands.
